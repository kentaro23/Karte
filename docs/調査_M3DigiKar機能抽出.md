# M3 DigiKar（エムスリーデジカル）競合ベンチマーク詳細レポート

調査対象: クラウド電子カルテ「M3 DigiKar」デモ環境（https://demo.digikar.jp、デジカル内科クリニック、内科・無床診療所想定）
調査手法: 全画面踏破＋実データE2E（赤坂太郎00001で受付→診察→処方→安全チェック→病名→会計まで実走）。設定/権限/マスタの変更・レセプト提出・会計確定など不可逆操作は「ボタン・ダイアログの存在確認のみ」で停止。

---

## 1. 製品概要・全体像

- **位置づけ**: M3（エムスリー）グループのクラウド型電子カルテ。診療所（クリニック）向け。SPA（React、`reception_karte/index-*.js`）。レセコンは別体（**WebORCA/ORCA連携**が前提、設定で「一体型」も選択可）。
- **アプリ二層構成**:
  1. **受付画面**（`/reception/{yyyymmdd}`）＝日次の患者ボード／受付・会計・レセプトの起点。
  2. **カルテ画面**（`/karte/patients/{patientId}/{date}?visitId=...`）＝3ペイン（左:患者情報タブ群／中央:2号紙SOAPエディタ／右:オーダ入力）。
  3. **施設管理画面**（`/institution_admin/*`）＝設定・マスタ・利用者・連携・帳票・統計。
- **アーキテクチャ要点**: クラウド本体 ↔ **院内エージェント（Agent V2）** ↔ {WebORCAレセコン, オンライン資格確認端末}。トークン認証（1時間有効）。患者アプリ**デジスマ（digisma）**連携、外部**検査会社 約170社**連携、M3独自の**RME（診療支援/疾患検出）**を内蔵。
- **API**: `https://demo.digikar.jp/public_api/*`（REST）。認証は`/public_api/jwt`・`/public_api/tokens/digisma`。エージェントは`/agent/dsps/channel/.../subscription/polling`でロングポーリング。

### 2-4. オーダ（右ペイン）

### 2-6. 安全チェック（保存時監査）
- **トリガ**: カルテ保存時に発火(設定「カルテ保存時処方監査確認:する」「算定チェック確認:する」)。
- **警告一覧ダイアログ**: カラム=処置行為・医薬品/種類/メッセージ。実観察した種類=**適応症チェック**「適応症がついていません」。行に削除/**追加**ボタン、下部「保存(強行)」、右上「**警告管理**」リンク。
- **適応症ダイアログ(追加押下)**: 左=薬剤の**添付文書 効能・効果**(アムロジン→1)高血圧症 2)狭心症＋注意文)。右=**データベース**(公式適応病名を確定/主病/疑いで即登録)＋**自動学習**(当院の併用実績から学習した推奨病名群:高尿酸血症/高コレステロール血症/不眠症…)＋傷病検索。確定すると病名自動付与＋警告解消(済バッジ)。
- **二次バリデーション**: 院外内服で**投与日数未入力**だと「カルテ保存エラー」で保存ブロック(該当行を赤表示)。
- **警告管理**(`/institution_admin/prescription_audit_filter`): 処置行為・医薬品/種類/レベル単位で警告を**無効化**でき、表示先(カルテ／要確認レセプト一覧)を制御。

## 3. 連携マップ（データフロー＆外部連携先）

**院内データフロー（実観察）:**
```
[受付] 患者検索/受付→visit生成
   │（受付visitがカルテ作成ダイアログに紐付く）
[カルテ作成] 保険/公費を選択しカルテ起票
   │
[診察] SOAP記載（主訴・所見）＋オーダ（処置・行為）
   │  ・処方/注射/検査=各マスタ実検索→行内で用法用量・院内外・日数
   │  ・診察料は自動算定（初診料291点を自動挿入）
   │
[保存] 監査発火 → 適応症チェック → 適応症追加で病名自動付与（傷病名タブに反映）
   │            → 算定/必須チェック（日数未入力で保存ブロック）
   │  ・病名自動送信ON → ORCAへ病名連携
[会計] ¥→会計ダイアログ（点→円、自己負担、領収書/明細/処方箋/薬剤情報/お薬手帳発行）→保存でステータス会計待/会計済へ
   │
[レセプト] 月次でレセプト作成（社保/国保、点検用→提出用、返戻・月遅れ処理）
```
**「何が何に波及するか」要点:**
- 受付ステータス（診察待→診察中→会計待→会計済）は受付ボード・カルテ・会計で共有。
- 処方の「適応症追加」＝病名登録（患者の傷病名に恒久反映）＋警告解消の二重作用。
- カルテ保存→（設定で）ORCAへ病名・診療行為が自動送信。
- 会計保存→領収書/明細/院外処方箋/薬剤情報/お薬手帳を一括発行。
- 文書雛形は患者情報・保険・病名・投薬・検査・バイタルを横断参照して差込。

## 4. E2Eで実観察した業務フロー（赤坂太郎 00001）

1. 受付ボード→患者リンクでカルテ遷移（`visitId=553818`）。当日カルテ無し。
2. 中央「＋」→**カルテ追加**(受付visit/保険無し/公費無し)で起票→オーダ入力が有効化。前回処方がDo候補として右ペインに出現。
3. 右**投薬**でアムロジン錠5mg検索→追加→数量1→**用法**[内]1日3回朝昼夕食後→院外。
4. 中央**主訴・所見**にSOAP記載(高血圧フォロー)。
5. **保存**→**警告一覧**「適応症がついていません(適応症チェック)」発火→**追加**→適応症ダイアログで**高血圧症を確定**(添付文書効能＋自動学習推奨を確認)→病名自動付与・警告解消。

## 5. UX観察（自社が学ぶ点／避ける点）

**学ぶべき優れた点:**
- **安全チェックの「警告→ワンクリック解決」体験**が秀逸。適応症未付与を検知→添付文書の効能・効果＋**自動学習(当院実績)推奨病名**を同一ダイアログで提示→確定一発で病名登録＆警告解消。クリック数最小で算定漏れ・査定リスクを潰す設計は最大の差別化。
- **病名登録の確定/主病/疑い ワンクリック＋修飾語プリ合成**。標準病名選択の摩擦が極小。
- **マスタ実検索の一貫性**（投薬/注射/検査いずれもキーワード→候補→行内編集）で学習コストが低い。用法・院内外・日数が行内完結。
- **過去カルテの縦並び/横並び二系統参照**＋セクション単位Do(↪)で再診効率が高い。
- **受付ステータスの色分け10段階**と行内インライン編集、表示カラム自由化＝受付の現場最適化が強い。
- **文書雛形のdocx/xlsx＋豊富な差込変数**で帳票自由度が高い(和暦/ローマ字/保険枝番/バイタルまで網羅)。
- **検査会社約170社＋POCT機器プリセット**は導入時の連携負担を大幅軽減。
- **RME(疾患検出デイリーレポート)**は他社にない受診勧奨/見落とし防止の付加価値。

**避けるべき/弱点(自社が上回る余地):**
- **内蔵統計が「月別受付回数」のみ**で貧弱。経営ダッシュボード(疾患別/医師別/点数推移/再診率/収益分析)が手薄→自社の差別化余地大。
- **検査結果(数値/基準値/グラフ/経過)がデモ非提供**で実機能を確認できず。標準実装＋トレンドグラフは要件で強化すべき。
- **レセコンが別体(ORCA依存)＋院内エージェント常駐**が前提。クラウド一体型・エージェントレス(完全SaaS)にできれば運用負荷で優位。
- **権限モデルが3ロール＋管理フラグのみ**と粗い。きめ細かな機能別権限(RBAC)で差別化可能。
- **SOAPが自由記載中心**で、構造化S/O/A/P・コード化アウトカム・テンプレ流し込みの作り込みは現場任せ。テンプレ/セットの初期コンテンツ充実とAI下書きで上回れる。
- マスタ更新がCSV/ORCA取込前提で、リアルタイム点数改定追従のUXは要検証。
