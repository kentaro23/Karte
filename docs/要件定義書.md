# Medixus カルテ（medixus-karte） 要件定義書

> 本書は「ゼロからの新規開発」ではなく、**既に稼働しているフルスタックEMR（Next.js 15 + Prisma）への増築（差分要件化）**を目的とする。
> 既存で実装済の領域（●）は「維持・本番化」要件として簡潔に確認し、一部実装（△）・未実装（×）を**要件化の中心**に置く。
> 競合「M3 DigiKar（エムスリーデジカル）」と厚労省標準（標準型電子カルテ／3文書6情報／3省2ガイドライン）に照らした4軸ギャップ分析（第3.5章）から、優先度付きの機能要件（第5章）を導出する。

## 第2章 背景・目的・スコープ

### 2.1 背景

- 国の到達目標として「遅くとも**2030年（令和12年）**に概ねすべての医療機関で電子カルテを導入、普及率約100%」が規制改革実施計画で明記された。診療所の電子カルテ導入率は約55%、特に200床未満が低く、ここが標準型電子カルテの主対象＝Medixus カルテの主戦場。
- 厚労省は「必要最小限の基本機能」を国が標準型として開発し、**差別化はオプション機能領域（＝現場ペイン解消・UX・連携の広さ）**に委ねる構図。Medixus カルテはこの差別化領域で勝負する。
- medixus-karte は既に**外来コアが貫通**している（受付→カルテ→SOAP→処方安全チェック→ロック→監査）。フロントは全28ルート・型チェック0エラー・本番ビルド成功。規制の背骨（追記専用＝真正性、ハッシュ連鎖監査、RBAC＋患者ACL、provenance厳格な安全エンジン）が**設計段階で内製済**であり、後付けが困難なこの部分を既に持つことが最大の参入優位。
- 一方で、(a) 病名×添付文書の適応症チェック（DISEASE_CONTRA）、(b) 検査結果のトレンド可視化、(c) 外部連携（オン資・レセコン/ORCA・電子処方箋・FHIR・PACS）、(d) 多くの△画面のバックエンド本番化と深掘り、が未達。これらが本書の要件化の中心。

### 2.2 目的

本書の目的は、medixus-karte を「M3 DigiKar と同等以上の現場価値」かつ「厚労省標準（174項目／3文書6情報／3省2GL）準拠」のプロダクトへ**増築**するための、実装契約となる要件（FR/IF/NFR/データ/移行）を、ID・入出力・業務ルール・受け入れ基準付きで定義することである。

### 2.3 スコープ

#### 2.3.1 診療形態スコープ

| 形態 | 本書での扱い | 現状 |
|---|---|---|
| 外来 | **主スコープ**（最優先・1号店実証ラボの対象） | コア貫通済（●多数） |
| 入院（中小病院） | 副スコープ（FE枠あり、深掘りはPhase 5） | 病床マップ・経過表・看護記録の枠は実装済（△） |
| 救急 | 副スコープ（`TriageLevel` 予約済、Phase 5） | データモデルのみ |
| 在宅・訪問診療 | 拡張スコープ（同一 `Encounter` 基盤拡張、Phase 6） | モデル enum（HOMECARE）のみ |
| 健診 | 拡張スコープ（Phase 6） | enum（CHECKUP）のみ |

#### 2.3.2 対象ロール（`JobType` enum と一致）

| ロール | enum | 主な利用機能 |
|---|---|---|
| 医師 | DOCTOR | カルテ記載・オーダ・処方確定・病名・カウンターサイン承認 |
| 研修医 | RESIDENT | カルテ記載・オーダ（指導医カウンターサイン対象） |
| 看護師 | NURSE | バイタル・看護記録・オーダ実施（注射三点認証）・経過表 |
| 薬剤師 | PHARMACIST | 処方監査・安全データレビュー昇格・調剤 |
| 事務（医事） | CLERK | 受付・予約・会計・レセプト・マスタ |
| （補助）検査技師/療法士/栄養士/管理者/システム管理 | TECHNOLOGIST / THERAPIST / DIETITIAN / MANAGER / ADMIN | 各部門業務・権限管理 |

#### 2.3.3 スコープ外（Won't / 本書では要件化しない）

- 歯科・調剤薬局（保険薬局）業務、DPC/PDPS の包括算定エンジンの内製（標準型レセコン＝WebORCA系／共通算定モジュール連携で代替）
- 医療機器（モダリティ）そのものの制御。PACS/LIS とは規格連携のみ
- AI による安全判断の自動生成（**設計原則として恒久的に禁止**。AIは下書き補助・要約等の非安全用途に限定し、provenance enum で物理担保）

### 2.4 前提（増築の不可侵原則）

1. **作り直さない**：既存のサーバーアクション署名・ドメイン状態機械・データモデルを土台に、内部を本番化し差分を足す。`handoff-cto.md` の「シグネチャを保ったまま内部実装を本番化」方針に従う。
2. **規制の背骨は不可侵**：追記専用（`db/triggers.sql`）、ハッシュ連鎖監査、provenance厳格・AI非生成、RBAC＋患者ACL は維持・強化のみ。
3. **連携は疎結合**：外部連携はすべて `packages/interop`（seam）配下に閉じ込め、レセコン依存部を疎結合に保つ。

### 3.2 M3 DigiKar 抽出と比較

M3 DigiKar（クラウド型・診療所向け・レセコンはWebORCA別体が前提）の機能を全画面踏破＋E2Eで抽出した。本書の比較に効く要点は以下。

#### 3.2.1 DigiKarの強み（学ぶべき＝Medixus が追随/凌駕すべき点）

1. **安全チェック「警告→ワンクリック解決」体験（最重要）**：カルテ保存時に「適応症がついていません」を検知→同一ダイアログで [添付文書の効能・効果] ＋ [公式適応病名DB] ＋ [自院実績からの**自動学習推奨病名**] を提示→確定/主病/疑い ワンクリックで病名登録＆警告解消。算定漏れ・査定リスクを最小クリックで潰す。**Medixus の DISEASE_CONTRA（Phase 2）はこの体験を実装する好機。**
2. **病名登録の摩擦最小化**：標準病名検索の各候補に「確定/主病/疑い」三連ボタン＋**修飾語プリ合成**（胃潰瘍→急性/慢性/出血性/術後…）＋特定疾患タグ。
3. **マスタ実検索の一貫性**：投薬/注射/検査いずれもキーワード→候補→**行内編集**（数量・用法・院内外・日数が行内完結）。検査は点数表示。診察料は保存時自動算定（初診料291点）。
4. **過去カルテ二系統参照**：診療履歴（縦並び＋種別フィルタ シ/診/薬/注/処/手/検/画/他/自/麻）＋カルテ一覧（日付横並び比較）＋セクション単位Do（↪）。
5. **受付ステータス10色＋行内インライン編集＋表示カラム自由化**（16項目ON/OFF）。
6. **文書雛形 docx/xlsx ＋豊富な差込変数**（和暦/ローマ字/保険枝番/未転帰傷病名/直近投薬/直近検査/バイタルまで）。
7. **二次バリデーション（保存ブロック）**：院外内服で投与日数未入力なら保存エラーで該当行赤表示。
8. **警告管理**：処置行為・医薬品/種類/レベル単位で警告を無効化、表示先（カルテ／要確認レセプト一覧）を制御。
9. **外部連携の広さ**：検査会社約170社＋POCT機器プリセット、患者アプリ デジスマ（予約/Web問診/キャッシュレス）、電子処方箋/一般名処方/リフィル、RME（疾患検出デイリーレポート）。

#### 3.2.2 DigiKarの弱み（Medixus が上回る余地＝差別化の主眼）

1. **内蔵統計が「月別受付回数」のみ**で貧弱。経営ダッシュボード（疾患別/医師別/点数推移/再診率/収益）が手薄。
2. **レセコンが別体（ORCA依存）＋院内エージェント常駐**が前提。Medixus は**クラウド一体型・エージェントレス（完全SaaS）**で運用負荷優位を狙える。
3. **権限モデルが3ロール＋管理フラグのみ**と粗い。Medixus は**RBAC＋個別override＋患者ACL**で既に上回る。
4. **SOAPが自由記載中心**。Medixus は **S/O/A/P 構造化（blocks）＋テンプレ初期コンテンツ＋AI下書き**で上回れる。
5. **データポータビリティ**が乗換コスト300〜500万円の障壁。Medixus は**FHIR/SS-MIX2 エクスポート＋共通データ移行レイアウト準拠**で「ロックインしない」を売りにできる。

### 4.2 連携アーキテクチャ方針

- **全外部連携は `packages/interop`（seam）に集約**し、サーバーアクションからは抽象IFで呼ぶ（レセコン依存部を疎結合化）。現状 stub 先行→本番実装。
- **クラウド一体型・エージェントレス志向**：DigiKarの院内エージェント常駐に対し、可能な限り完全SaaS（クラウド間連携）で運用負荷を下げる。ただしオン資・PACS等で院内端末/ゲートウェイが不可避な部分は明示。
- **標準規格準拠を接続条件とする**：オン資・電子処方箋・電子カルテ情報共有サービス（FHIR）・JAHIS検査/放射線・DICOM・SS-MIX2 を各IFのプロトコルとする（第6章）。
- **算定はレセコンに委譲**：点数算定・レセプト生成は標準型レセコン（WebORCA系）/共通算定モジュールへ委譲し、Medixus は診療情報・オーダ・病名・中途終了データの連携に徹する。

### 4.3 データ方針

- **追記専用・provenance・ハッシュ連鎖を全新規テーブルにも適用**（安全・診療・監査に関わるものは UPDATE/DELETE 禁止トリガ対象に追加）。
- **コード標準を内部正規化キーに**：病名=ICD-10対応標準病名（病名管理番号）、医薬品=YJ/HOT/レセ電、検査=JLAC10/11、文書=HL7 FHIR。マスタは `MasterVersion`/`ImportRun` で版管理。
- **6情報は FHIR リソースに正規化**：傷病名=Condition、感染症/検査=Observation、薬剤アレルギー等/その他アレルギー等=AllergyIntolerance、処方=MedicationRequest（第6.5章）。

### 5.5 処方安全（最重要差別化領域）

#### FR-RXSAFE-01 4大＋アレルギーチェック（維持・本番化）
- **優先度**：Must（維持） / **ギャップ**：G11 / **項番**：174:59-61 / **増築箇所**：`order-checks/engine.ts`, `rule-engine`

#### FR-RXSAFE-02 病名連動 適応症チェック＋ワンクリック病名登録（🎯最重要新規）
- **優先度**：Must / **ギャップ**：G11,G14 / **増築箇所**：`order-checks/engine.ts`（DISEASE_CONTRA追加）, `DrugIndication`(実装済), `PatientDiagnosis`, 新規「適応症追加ダイアログ」
- **説明**：DigiKarの差別化機能を実装。処方保存時、各薬剤の適応症（`DrugIndication.icd10Codes`/`indicationText`）と患者の有効病名（`PatientDiagnosis`）を突合し、**適応症未付与を検知**。検知時に「適応症追加ダイアログ」を表示し、[添付文書の効能・効果（`DrugIndication`）] ＋ [公式適応病名から確定/主病/疑い ワンクリック登録] を提示。確定で `PatientDiagnosis` を自動追加（開始日=当日）し、当該警告を解消（済）。

### 10.2 戦略的優先順位（投資の主眼）

1. **🎯 適応症ワンクリック＋病名（FR-RXSAFE-02 / FR-DX-01）**：DigiKar最大の差別化機能。Medixusは `DrugIndication`（実装済）×`PatientDiagnosis` の突合で実装可能＝**最小投資で最大の競争力**。Phase 2 最優先。
2. **制度MUST連携（Phase 7）**：オン資・電子処方箋・FHIR 3文書6情報・レセコンは2030年普及目標・診療報酬加算評価に直結＝**事業継続条件**。`interop` seam に集約。
3. **検査結果トレンド（FR-LAB-01）＆経営ダッシュボード（G28）**：DigiKarの弱点領域＝**明確な凌駕ポイント**。
4. **データポータビリティ（IF-EXT-07）**：乗換障壁（300-500万円）を逆手に「ロックインしない」を売りに。
